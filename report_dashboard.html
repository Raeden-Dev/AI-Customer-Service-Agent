Internal Architecture Report

AI Customer Service Agent

Generated 2026-07-20 App version 1.0.0 11 sections 26 Java classes · 3 Python modules ~2,359 lines of code

This is a voice-driven customer-service agent built from two cooperating runtimes. A Spring Boot (Java 21) application is the brain and control plane: it runs an operator terminal (agent>), holds all state through four managers, persists everything as JSON, and exposes a small REST API on localhost:8080/api. A Python subprocess is the ears and mouth: it captures microphone audio, transcribes speech via Google STT, generates replies via Google Gemini, and streams every utterance back to Java over HTTP. When you start a call, Java spawns the Python process; when you end it, Java kills it and archives the conversation. The single most important thing to know if you want to read the live conversation is that every customer line and every AI line converges on one method — CallManager.handleChatMessage() — which is your primary interception point (see Reading Dialogs).

01 — System Overview

Two runtimes, one conversation

Java owns state, control, and persistence. Python owns audio, speech-to-text, and the LLM. They meet only over a handful of REST calls.

Key Takeaways
  • Java = control plane. Spring Boot app with an interactive operator CLI, four managers, and a 6-route REST API.
  • Python = media plane. A short-lived subprocess launched per call; does mic capture, Google STT, and Gemini replies.
  • Java drives Python. CallManager spawns stt_sender.py on start-call and terminates it on end-call.
  • Python pulls config, pushes dialog. It GETs config & AI settings, then POSTs every recognized line back to Java.
  • Everything is a JSON file. No database — businesses, settings, call history, and transcripts all live under data/.

03 — Class Map

How the classes wire together

Spring injects the managers; the operator terminal and the controllers both call into them; Python reaches the same managers from outside over REST.

Key Takeaways
  • Main boots Spring, then hands the context to ConsoleTerminal, which pulls all four managers out.
  • Controllers are thin: each just forwards to one manager method — no logic of its own.
  • AISettingsManager and CallManager both depend on BusinessManager to know the active business.
  • Python is wired in only through the four REST controllers — never directly to a manager or model.

04 — Managers

The four managers & what they hold

Each manager is a Spring @Component — one responsibility, its own slice of state, and a small public surface.

Key Takeaways
  • ConfigManager — one global Config; loads/saves config.json; sets the console debug level.
  • BusinessManager — the registry of businesses and which one is active; everything else keys off this.
  • AISettingsManager — per-business AI persona, cached in memory, backed by ai-settings.json.
  • CallManager — the live-call engine: starts/ends calls, owns the Python subprocess, records every message.

ConfigManager 40 LOC

holds: Config config · @PostConstruct init()
  • reload() → load/create data/config.json, apply debug level
  • getConfig() → Config
  • updateConfig(Config) → replace + persist + re-apply level

BusinessManager 101 LOC

holds: Map<name,Business> · Business activeBusiness
  • reload() → scan businesses/*/business.json
  • getBusiness(name) → Business (case-insensitive)
  • listBusinesses() → List<Business>
  • addBusiness(name, details) → new UUID + save
  • removeBusiness(name) → boolean
  • save(Business) → write business.json
  • getActiveBusiness() / setActiveBusiness(name)

AISettingsManager 66 LOC

holds: Map<name,AISettings> cache · needs BusinessManager
  • reload() → clear cache, load active settings
  • getActiveSettings() → AISettings (or defaults)
  • getSettings(name) → cache → loadOrCreate
  • updateActiveSettings(s) → cache + write ai-settings.json

CallManager the live-call engine · 295 LOC

holds: activeCall · callBusiness · Process sttProcess · Map<name,CallHistory>
  • startCall(agent, customer, type) → new Call + launch STT subprocess
  • endCall() → stop STT, save history + transcript.txt
  • handleChatMessage(Role, content) ★ records every user/AI line
  • handleTranscript(text, translated, isFinal) → set final transcript
  • isCallActive() / getActiveCall() / getActiveHistory()
  • private: launchSttProcess · stopSttProcess · pipeSttOutput · saveCallTranscript · loadHistory
★ The one method to remember CallManager.handleChatMessage(role, content) is where every recognized customer line and every generated AI line lands in real time. It appends to the call's transcript and prints to the operator console. If you want a live feed, a websocket, a UI, or analytics — hook here first. See Reading Dialogs.

06 — REST API

The 6 endpoints on localhost:8080/api

The entire Java⇄Python contract. Python fetches with GET, reports with POST. Every controller delegates to exactly one manager method.

Key Takeaways
  • Python reads behavior via GET /config and GET /ai-settings at the start of each call.
  • POST /chat-message is the live per-utterance channel — the busiest and most interesting route.
  • POST /transcript fires once at call end with the full session text.
  • Base URL is hard-coded in Python's config.py as http://localhost:8080/api.

07 — Live Call Flow

The life of a call, end to end

From start-call to end-call — and the exact path a single spoken sentence takes to become an AI reply.

Key Takeaways
  • start-call creates a Call and spawns python -u stt_sender.py as a child process.
  • Python transcribes speech (Google), optionally switches language, then asks Gemini for a reply.
  • Both the user line and the AI line are POSTed back individually to /api/chat-message.
  • end-call sends SIGTERM to Python, which flushes a final transcript; Java then writes the archive.

The full round trip, including config fetch, the per-phrase loop, and teardown. Java only learns the final flat transcript at the end — the granular turn-by-turn record is built live via chat-message.

Detail · language switching Inside the loop, Python checks each utterance against supportedLanguages aliases (e.g. saying "bangla" or "বাংলা"). On a match it restarts the recognizer in the new locale and emits please-wait / ready system messages — all of which also flow through /api/chat-message.

08 — Reading Dialogs · what you asked for

Where to intercept the live conversation

Seven concrete places you can tap the User↔AI dialog while a call is happening, ranked from best to most specialized.

Key Takeaways
  • Best single hook: CallManager.handleChatMessage() — every line, both roles, in real time, in-process.
  • Best HTTP hook: POST /api/chat-message (or a Spring filter on it) if you want to tap from outside Java.
  • At the source: Python handle_phrase() already holds the user text and the AI answer before sending.
  • Read-only tap: the live Call.getMessages() list, or Console.getSessionLogs(), can be polled.
  • /api/transcript is not live — it only carries the full text once, at call end.
★ Primary recommendation Add your listener in CallManager.handleChatMessage(role, content) (java-backend/…/managers/CallManager.java:128). Every USER and AI utterance passes through it before it's stored or printed. Broadcast from there (WebSocket / SSE / event bus) and you get the whole live conversation with zero changes to Python.

Interception points colored by nature: violet = primary hub, emerald = source-side, blue = read-only, amber = end-only. Points 1–4 give you the conversation live; 5–6 let you observe it; 7 is the after-the-fact summary.

Caution · common mistake Don't rely on POST /api/transcript for live monitoring. handleTranscript() only acts when isFinal == true, which Python sends once, during teardown. For anything real-time, use points 1–4.
Opportunity · minimal live-feed recipe In handleChatMessage(), after activeCall.addMessage(...), publish the same ChatMessage to a Spring @EventListener or a WebSocket topic. One added line gives every external consumer the live transcript — the model and endpoints already carry role + timestamp.

09 — Configuration Reference

Every knob, and where it's read

Global behavior lives in Config (mirrored to Python); the AI persona lives per-business in AISettings.

Key Takeaways
  • STT sensitivity is four tunable doubles: pause / phrase / non-speaking / ambient-noise thresholds.
  • allowAiResponse is a kill-switch — set false to transcribe without generating replies.
  • Adding a spoken language = one entry in supportedLanguages with aliases + prompts.
  • The AI persona is assembled by Python's build_prompt() from role + reply + org instructions.

10 — Data & Storage

How state is persisted

No database. Gson reads and writes JSON files; every write goes through FileUtils, every path through PathUtils.

Key Takeaways
  • Each business is a self-contained folder holding its settings, call history, and transcripts.
  • Ending a call writes twice: the structured call-history.json and a human-readable <callId>.txt.
  • FileUtils.updateJsonFile() deep-merges new default keys into existing files without losing data.
  • Errors are captured to logs/errors/error_<at>_<id>.log with full stack traces.

11 — Extending It

Where to build next

Grounded starting points based on scaffolding that already exists in the code.

Key Takeaways
  • Live UI / feed: broadcast from handleChatMessage() — the model already carries role + timestamp.
  • Analytics: populate the dormant sentiment / callContext fields on Call.
  • Customer context: the Instructions enum + Organization model are pre-built but unwired.
  • Harden config: pull the API key out of the served JSON before any deployment.
Opportunity · real-time UI Tap point #1, push each ChatMessage to a WebSocket, and render a live agent-assist screen. No Python changes needed.
Unique · unwired scaffolding The Instructions enum (NEW_CUSTOMER, EXISTING_CUSTOMER, WRONG_NUMBER…) and the Organization model exist but aren't referenced — ready to drive smarter prompts.
Detail · reserved fields Call.sentiment, Call.callContext, and Call.translated are serialized but never set — safe slots for new pipelines.
Risk · before shipping Externalize the Gemini key, add auth to the REST API, and consider persisting per-turn messages incrementally rather than only at endCall().
