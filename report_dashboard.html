AI Customer Service Agent

Project guide · version 0.2 · updated 2026-07-21

A voice-based customer service agent. A caller speaks into the microphone, the app turns the speech into text, an AI (Google Gemini) writes a reply using the business's own knowledge files, and the reply is spoken back out loud. Each business has its own folder of JSON files — its settings, its knowledge, its clients, and its call history. No database is needed.

What this project does

Two programs work together. Java is in charge; Python does the audio work.

☕ Java backend (java-backend/)

  • Gives you the agent> console where you run commands
  • Stores everything as JSON files under data/
  • Runs a small REST API on localhost:8080 that Python talks to
  • Starts and stops the Python process when a call starts and ends
  • Saves the call history and a readable transcript after every call

🐍 Python scripts (python-scripts/)

  • Listens to the microphone and turns speech into text (Google)
  • Builds the AI prompt and asks Gemini for a reply
  • Speaks the reply out loud (text-to-speech, works offline)
  • Understands spoken language switching (English ⇄ Bangla)
  • Sends every line of the conversation to Java so it gets saved
Main idea: the AI only knows what you put in the business's JSON files. Edit intelligence.json (business knowledge) and clients.json (client records), and the AI uses them on the very next call.

How to run it

You need Java 21+, Maven, Python 3.10+, a microphone and speakers.

  1. Install the Python packages (one time): 
    cd python-scripts
python -m venv .venv
.venv\Scripts\pip install -r requirements.txt
  2. Add your Gemini API key in java-backend/data/config.json → "apiKey".
  3. Start the backend (it launches Python by itself during calls): 
    cd java-backend
mvn spring-boot:run
  4. Make a test call at the agent> prompt: 
    use test1          (pick the sample business)
clients            (see the sample clients)
start-call C001    (call as client Rahim — or just start-call for a new customer)
... speak into your mic, the AI answers out loud ...
end-call

The four call modes

Every call is in exactly one mode. The mode decides which extra instructions the AI gets.

1 · New Customer

starts with: start-call

The caller is unknown. The AI greets them, asks for their name and what they need, and explains services using the business knowledge.

2 · Existing Customer

starts with: start-call <client_id>

The caller is a known client. The AI greets them by name and uses their notes and past issues from clients.json to give personal help.

3 · Wrong / Scam Number

AI detects it, or: set-mode wrong_number

The AI stays polite, shares no business or client details, explains they may have the wrong number, and ends the call.

4 · Complex Request

AI detects it, or: set-mode complex_request

Too hard for the AI (refunds it can't approve, legal issues, escalations). It says a human agent will take over — then goes quiet while the app keeps transcribing. The hand-over is simulated.

How the AI switches mode by itself: the prompt tells the AI it may end a reply with [MODE:WRONG_NUMBER] or [MODE:COMPLEX_REQUEST]. Python removes the tag before speaking, and reports it to Java (POST /api/call-mode). Java records the switch in the call log, and the next AI reply follows the new mode's instructions.

How a call works, start to finish

What happens when you type start-call C001, talk, and type end-call.

sequenceDiagram autonumber participant You as You (console) participant Java as Java backend participant Py as Python participant AI as Gemini AI You->>Java: start-call C001 Java->>Py: starts the Python process Py->>Java: fetches settings + call context (mode, business info, client info) Py->>You: AI speaks the greeting out loud loop every sentence you say Py->>Py: microphone → text (Google speech-to-text) Py->>Java: sends your words (saved in the call log) Py->>AI: prompt = persona + mode + business info + client info + conversation so far AI-->>Py: reply text (may end with a [MODE:...] tag) Py->>Java: sends the AI reply (and any mode switch) Py->>You: speaks the reply out loud end You->>Java: end-call Java->>Py: stops the Python process Java->>Java: saves call-history.json + transcript .txt
Why the AI never talks to itself: speaking and listening share one thread, so the microphone is not listening while the AI's voice is playing.

The data folder

Every business is one folder under java-backend/data/businesses/. These files are the whole "database".

java-backend/data/
├── config.json                global settings
└── businesses/test1/
    ├── business.json          id, name, contact info
    ├── ai-settings.json       AI persona
    ├── intelligence.json      business knowledge ★
    ├── clients.json           client records ★
    ├── call-history.json      every finished call
    └── transcripts/
        └── <callId>.txt       readable transcript
★ = the two files you edit to make the AI smarter. They are re-read at the start of every call — edit, then start-call. No restart needed.
FileWhat's inside
intelligence.json about (one paragraph), services (list with prices), policies (list), faqs (question + answer pairs)
clients.json One record per client: clientId (used in start-call C001), name, phone, email, accountType, notes, pastIssues (list)
ai-settings.json Which Gemini model to use, who the AI is (roleInstructions), how it should answer (replyInstructions)
call-history.json Every finished call with its mode, caller id, times, messages and transcript
New businesses get starter template files automatically the first time they are used, so you always have the right shape to fill in.

All in java-backend/data/config.json. Python re-reads them at the start of each call.

The voice comes from your operating system (no internet needed). Run python tts_speaker.py inside python-scripts to list the installed voices and hear a test sentence.
Before sharing this project anywhere: your real API key sits in plain text in data/config.json and is also returned by GET /api/config. That is fine on your own machine, but move it to an environment variable before deploying.

REST API

How Python and Java talk, on http://localhost:8080/api. You can also call these yourself (e.g. from a future web UI).

MethodPathUsed for
GET/api/configPython fetches the global settings at call start
POST/api/configOverwrite the global settings
GET/api/ai-settingsThe AI persona of the active business
POST/api/ai-settingsUpdate the persona
GET/api/call-contextEverything about the current call: mode + instructions, business knowledge, client profile, greeting (new)
POST/api/call-modePython reports a mode switch the AI flagged (new)
POST/api/chat-messageEvery live line of the conversation (user / ai / system)
POST/api/transcriptThe full transcript, sent once when the call ends

What's new in v0.2

Compared to v0.15 (speech recognition + Java listener).

  • The AI talks now. Replies and the opening greeting are spoken out loud (offline text-to-speech, configurable speed/volume/voice).
  • Business knowledge files. intelligence.json per business — about, services, policies, FAQs — used by the AI on every call.
  • Client records. clients.json per business; start-call C001 makes the AI treat the caller as that client.
  • Four call modes — New Customer, Existing Customer, Wrong/Scam Number, Complex Request (simulated hand-over to a human).
  • Conversation memory. The AI now remembers the current call's dialog, so follow-up questions work.
  • New commands — clients, intel, set-mode; and start-call takes a client id.
  • Cleaner code. All REST controllers now live in one api package; dead code removed; comments everywhere; README rewritten.

Ideas for later

  • Web dashboard: a page that shows the live conversation — everything already flows through CallManager.handleChatMessage(), so that is the place to broadcast from.
  • Real phone calls: swap the microphone for a telephony service (e.g. Twilio) while keeping the Java side unchanged.
  • After-call notes: ask the AI to summarize each finished call and append it to the client's pastIssues automatically.
  • Sentiment: the Call model already has an empty sentiment field waiting to be filled.
  • Security: move the API key to an environment variable and add authentication to the REST API before any deployment.
